
## Invesco Salesforce App — Architecture Decision (03:30 UTC)

**Key constraints confirmed by Nathan:**
1. **Apex is a thin API client** — NOT reimplementing business logic. Calls existing ForwardLane Backend (Django) + Signal Builder Backend (FastAPI) on Railway
2. **No Agentforce** — Invesco doesn't use it
3. **No Data Cloud** — Invesco doesn't use it  
4. **Pure Apex + LWC** — clean native SF UI shell with Named Credential callouts to Railway APIs
5. **No custom objects needed** — all data lives in Railway PostgreSQL databases
6. **Auth: Per-Org Named Credential** for demo, Per-User/JWT Bearer for production

**API surface being wrapped:**
- Easy Button APIs: dashboard, clients, signals, actions, meeting-prep, nl-query
- Signal Builder APIs: signal CRUD, validation, nodes, results, schema
- Core Admin APIs: clients, CRM notes, holdings, transactions, data sources, jobs
- AI APIs: content recommendations, similar documents
- Analytics APIs: dashboard, advisors, signals
- Enrichment APIs: single/batch lead enrichment

**Timeline: ~10 days** (down from 14 since no custom objects or data sync needed)
